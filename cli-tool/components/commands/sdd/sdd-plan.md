# SDD Plan

Generate technical implementation plan: $ARGUMENTS

## Instructions

Translate the functional spec into a concrete technical plan. This is where you define HOW — tech stack, architecture, data model, and contracts.

### Step 1: Validate Prerequisites

```bash
BRANCH=$(git branch --show-current)
```

Verify `$BRANCH` matches `NNN-feature-name`. Load from the feature directory:

- **Required**: `specs/$BRANCH/spec.md`
- **Required**: `CONSTITUTION.md`

Check that `spec.md` has no `[NEEDS CLARIFICATION]` markers remaining. If any exist, STOP and instruct user to run `/sdd.clarify` first.

### Step 2: Extract Tech Stack

Parse `$ARGUMENTS` for:
- Language and runtime (e.g., TypeScript/Node.js, Python 3.12, Go 1.22)
- Frameworks (e.g., Express, FastAPI, NestJS, Django)
- Storage (e.g., PostgreSQL, DynamoDB, SQLite, S3)
- Testing framework (e.g., Jest, pytest, Go test)
- Target platform (e.g., AWS Lambda, Docker, local CLI)
- Any constraints mentioned (e.g., "no ORMs", "serverless only")

If `$ARGUMENTS` is empty, ask the user to specify their tech stack before continuing.

### Step 3: Constitution Check

Review `CONSTITUTION.md` and verify the proposed stack and approach comply with all principles.

If a principle is violated:
- Document the violation in a `Complexity Tracking` table in plan.md
- Either adjust the approach to comply, or justify the exception explicitly

**This check is a GATE — do not proceed if there are unjustified constitution violations.**

### Step 4: Phase 0 — Research

For each technology choice where there's uncertainty or rapid change (new frameworks, specific version behaviors, integration patterns), create `specs/$BRANCH/research.md`:

```markdown
# Research: [FEATURE NAME]

## [Technology/Decision Area]

**Decision**: [What was chosen]
**Version**: [Specific version if relevant]
**Rationale**: [Why this choice]
**Alternatives considered**: [What else was evaluated and why rejected]
**Key constraints**: [Important limitations or gotchas]
**References**: [Docs/links if relevant]

---
```

Research should resolve any technical unknowns from the spec. Use web search for rapidly evolving technologies.

### Step 5: Phase 1 — Design Artifacts

#### 5a. Write plan.md

Create `specs/$BRANCH/plan.md`:

```markdown
# Implementation Plan: [FEATURE NAME]

**Branch**: `NNN-feature-name` | **Date**: YYYY-MM-DD | **Spec**: specs/NNN-feature-name/spec.md

## Summary

[One paragraph: primary requirement + technical approach]

## Technical Context

**Language/Runtime**: [e.g., TypeScript 5.x / Node.js 22]
**Framework**: [e.g., Express 5, NestJS 10]
**Storage**: [e.g., PostgreSQL 16 via pg, DynamoDB on-demand]
**Testing**: [e.g., Jest + Supertest]
**Platform**: [e.g., AWS Lambda + API Gateway, Docker Compose]
**Performance Goals**: [from spec Success Criteria, translated to technical targets]
**Constraints**: [from constitution + $ARGUMENTS]

## Constitution Check

| Principle | Status | Notes |
|-----------|--------|-------|
| [Principle I] | ✅ Compliant / ⚠️ Exception | [notes] |

## Architecture

[Describe the overall approach: layers, key components, data flow]

## Project Structure

```text
src/
├── models/         — Domain entities

├── api/            — HTTP handlers / Lambda functions

tests/
├── unit/
└── integration/
```

## Key Technical Decisions

- **[Decision]**: [Rationale] (alternative considered: [X], rejected because [Y])

## Complexity Tracking

> Fill only if constitution violations require justification

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|

## Artifacts Generated

- `research.md` — Tech stack decisions
- `data-model.md` — Entity definitions
- `contracts/` — Interface contracts (if applicable)
- `tasks.md` — Generated by /sdd.tasks
```

#### 5b. Write data-model.md (if feature involves data)

Create `specs/$BRANCH/data-model.md`:

```markdown
# Data Model: [FEATURE NAME]

## Entities

### [EntityName]

**Purpose**: [What this represents]

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| id | UUID/string | required, unique | Primary identifier |
| [field] | [type] | [constraints] | [description] |

**Relationships**:
- Has many [OtherEntity] via [field]
- Belongs to [ParentEntity]

**State transitions** (if applicable):
- draft → active → archived

---
```

#### 5c. Write contracts/ (if feature has external interfaces)

For web services, CLIs, or libraries — create `specs/$BRANCH/contracts/`:

**REST API** → `contracts/api-spec.md`:
```markdown
# API Contract: [FEATURE NAME]

## POST /resource

**Purpose**: [What this endpoint does]

**Request**:
```json
{
  "field": "string (required)"
}
```

**Response 200**:
```json
{
  "id": "uuid",
  "field": "string"
}
```

**Error responses**: 400 (validation), 401 (auth), 404 (not found), 500 (server error)
```

Skip contracts if purely internal feature (build scripts, CLI tools without public API, etc.)

### Step 6: Update SDD Context

Update `.claude/sdd-context.md`:
```
- **Plan**: specs/NNN-feature-name/plan.md ✅
- **Research**: specs/NNN-feature-name/research.md ✅
```
